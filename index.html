How To Read This Dashboard

Use this section when comparing your observed slow peaks with the captured data.

Start with the timeline. Keep Sample gap, Scheduler delay, Monitor loop, System query, Process query, and UI probe enabled. Tall spikes here mean the machine was slow to schedule normal work, process UI messages, or answer Windows process/performance queries.

Click a spike or annotation. The Selected Time panel shows the nearest raw sample. Nearby Processes shows what was active around that point; Events shows warnings/errors near the same time.

Manual annotations matter. A manual red band is when you personally felt the machine was unusable. If it lines up with sample gaps, query stalls, or UI delay, the monitor captured the same symptom.

Current hypothesis for the 11:09 run: this does not look like RAM, CPU, or disk saturation. The strongest signal is Windows management/performance query stalling, possibly caused by a lower-level driver, security/VPN layer, or vendor service.

Ruled out for this run: Windhawk was not running. Its service was stopped at the detected spike times, so it should not be treated as a live suspect for this dataset.

Most useful next tests: compare one Chrome Incognito/extensions-disabled run and one VPN-disconnected run if possible. If one test removes the spikes, that component becomes the likely owner.

Useful filters: try chrome, MsMpEng, MsSense, SSLVpnClient, IDBWM, WmiPrvSE, and logioptionsplus.
